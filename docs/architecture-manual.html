Modules
cli.py 提供來源、URL evidence、品質、freshness、run、foundry 與 review 工作台操作；後者只在 127.0.0.1 提供單使用者介面，自動匯入 candidate、與 current／baseline 比對、保存結構化 handoff，並只在人工明確核准後更新 current。它不呼叫模型，也不取代 validation。GitBook 快取與草稿 facts 仍是本機、非權威的擷取輔助。 檔案 I/O 出口：generate/、run/、diff/、preparation/、score/、source_quality/、shadow/report.py、URL corpus 快取、foundry/ 與透過 Foundry 寫入的 review/workflow.py 負責寫入。adapters/fragments.py 是把來源內容 materialize 成 exact fragment 的 read-side I/O exit；Core 與 Domain 維持 pure。
Core shadow integration：assemble --architecture-mode shadow 會在 legacy validation 後呼叫 compatibility-only 的 shadow/ 套件。Pure bridge 把 manifest/plan 映成 Core evidence 與 claims，in-memory runner 只執行到 Core validation，且只有 shadow/report.py 會寫入觀測性的 <run-dir>/core/ 產物或 core/error.json。Legacy validation、score、approval、Foundry、status 與退出碼仍具權威。 選填 v1 evidence[] 會在建立 run 前透過 adapters/fragments.py 重新 materialize、驗證 source identity／locator／fragment digest，並以 shared plan projection 解析 claim path。Claim path 以 explicit_support、derived_support、contradicts 或 insufficient relationship 連到 exact typed fragment。沒有 parsed scalar 的 prose 使用 fail-closed 的單一 claim binding；OpenAPI structural derivation 限 versioned allowlist，且 Core 會連同每個必要的 local-reference context fragment 重新計算與驗證 digest chain。整份文件級 legacy citation 維持 unverified。relationships.json 與 projections/provenance.json 保存 claim → relationship → fragment → artifact 鏈。
