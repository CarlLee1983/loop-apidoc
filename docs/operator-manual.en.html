Architecture boundary: this manual documents the current agent-native CLI compatibility adapter. The stable product core is model/platform independent; see Architecture.

Loop Engineering's source-grounded API documentation pipeline

繁體中文版見 README.md。

loop-apidoc is a repeatable CLI that turns API integration docs of varying format and completeness into consistent, traceable, standardized artifacts:

  • OpenAPI 3.1 YAML (openapi.yaml)
  • Traditional-Chinese Markdown integration guide (api-guide.zh-TW.md)
  • Offline manual review page (review.html)
  • Source provenance data (provenance.json)
  • Validation and gap report (validation/report.{json,md})

Core principle: the source documents are the only source of truth. Anything a source does not state is never inferred; when required information is missing, validation fails and lists the gaps explicitly rather than filling them with conventions.

How it works

The extraction engine is the current coding agent itself. Inside a Claude Code plugin or OpenAI Codex CLI session, the agent follows skills/loop-apidoc/SKILL.md to read the sources and extract via a read-only subagent fan-out (each subagent only reads files and searches, returning JSON without writing files); the orchestrating agent writes the results into inventory.json + endpoints/*.json, then calls the deterministic CLI assemble to run the back half of plan → generate → validate.

The CLI itself does not create sessions, log in to private sites, or drive a browser — it only assembles and validates after the agent has written the extraction JSON.

An earlier run-agent CLI mode that extracted via the claude -p subprocess was retired in 2026-06 (together with the NotebookLM extraction backend); agent-native is now the only extraction path.

Full pipeline

preprocess (optional) → URL navigation/caching (optional: catalog-url → select-url → cache-url-pages) → assess-sources quality gate (optional) → extraction (agent read-only subagent fan-out) → verify-extraction → manifest → normalization plan → generate (OpenAPI + Markdown) → validate → correction (agent-driven)

When validation fails, assemble reports the classified issues via --json; correction is agent-driven: guided by the report, the agent re-reads the relevant sources, overwrites the corresponding extraction JSON, and re-runs assemble, for up to 3 rounds. If it still fails after three rounds, or it hits a gap/conflict that cannot be fixed, it reports the gap/conflict instead of filling it in.

Using it with the OpenAI Codex CLI

The same skill can also run under Codex. Codex does not set ${CLAUDE_PLUGIN_ROOT}, so install the CLI as a global command and mount the skill into Codex's skills directory:

# 1. Install the CLI as a global loop-apidoc command
uv tool install --from /path/to/loop-apidoc loop-apidoc

# 2. Mount the skill into Codex (a symlink is enough; edits sync automatically)
ln -s /path/to/loop-apidoc/skills/loop-apidoc ~/.codex/skills/loop-apidoc

SKILL.md auto-detects the environment via the <APIDOC> placeholder: with $CLAUDE_PLUGIN_ROOT it uses the plugin's bundled CLI, otherwise it falls back to the global loop-apidoc. The rest of the flow (extraction → assemble → validate → correct) is identical on both sides.

Installation

Requirements: Python >=3.11, with environments managed by uv.

# Install dependencies
uv sync

# Confirm the CLI runs
uv run loop-apidoc --help

Once installed as a Claude Code plugin, it is usable inside a Claude session with the CLI bundled in the plugin; for Codex, see "Using it with the OpenAI Codex CLI" above.

Providing sources

Before the agent extracts, the operator only needs to have the sources ready:

  1. Put local source files into a source directory (referred to as <SOURCES>).
  2. Pass public URLs directly with --url; the agent fetches them as text.
  3. For table-heavy or large PDFs, first convert them to high-fidelity markdown with preprocess (see below), then point the extraction at the output directory.

No external Notebook to create, no browser login needed — source reading is done by the current agent within the session.

Supported source formats

PDF, Markdown, Microsoft Word, OpenAPI JSON/YAML, public URLs.

Usage

The CLI provides source processing, URL evidence, draft extraction, quality and extraction, scheduled source freshness, run processing, Foundry asset governance, and the local human-review review workbench. The day-to-day extraction flow is driven by the skill; each command can also be used standalone.

preprocess — convert PDF to high-fidelity markdown (optional)

uv run loop-apidoc preprocess --sources ./sources --out ./work/sources_md

Uses pymupdf4llm to convert each PDF in the --sources directory into markdown that preserves table and heading structure (non-PDF text sources are copied as-is). --sources can also name one file, in which case only that file is processed. Converting table-heavy or large PDFs before extraction avoids the table distortion of reading the raw PDF; afterwards, point the extraction subagents at the --out directory.

manifest — build the source manifest

uv run loop-apidoc manifest --sources ./sources-or-file [--url <URL> ...] [--output manifest.json]

Scans local sources, recording relative path, format, size, SHA-256, scan time, whether supported, duplicate determination, and processing status; public URLs additionally record fetch time, HTTP status, and content hash. --sources accepts a directory or one source file; for a file, its parent becomes sources_root and the manifest contains only that file. When --output is omitted, output goes to stdout.

URL navigation and caching — catalog-url / select-url / cache-url-pages / related-url-pages

# Download the entry page only once and build a sidebar navigation index (does not auto-fetch child pages)
uv run loop-apidoc catalog-url --url <entry page URL> --output ./work/url_sources/catalog.json

# Select catalog nodes by explicit scope (downloads no URLs)
uv run loop-apidoc select-url --catalog ./work/url_sources/catalog.json --branch <branch> --term <keyword> --output ./work/url_sources/selection.json

# Cache all catalog pages and build local body/link/entity indexes (no model involved)
uv run loop-apidoc cache-url-pages --catalog ./work/url_sources/catalog.json --output ./work/url_corpus

# Emit candidate page cards from body links and shared entities; the model only reads a candidate's body when needed
uv run loop-apidoc related-url-pages --corpus ./work/url_corpus/corpus.json --url <page URL> --output ./work/candidates.json

A token-efficient web-doc reading flow: catalog.json is the complete navigation coverage universe; corpus.json embeds no body text, only pointing at the local raw/ and body/ files. When the catalog is empty or the document is a single page, use cache-url-entry --url ... --output ... to cache the entry page directly; already-downloaded HTML can be converted to supported Markdown with normalize-html-snapshot --input page.html --url ... --output sources/page.md, which also writes a .source.json provenance sidecar carrying the original URL and SHA-256.

snapshot-openapi-url — pin a directly-linked OpenAPI source

uv run loop-apidoc snapshot-openapi-url --url <OpenAPI JSON/YAML URL> \\
  --sources ./sources --coverage ./work/url_sources/coverage.json [--confirmed-by-user]

For cases where a URL already returns Swagger 2.0 or OpenAPI 3.x JSON/YAML directly. The command validates the spec declaration, pins the raw bytes as a local source, outputs the SHA-256, and writes a coverage ledger with method: direct. An existing snapshot and coverage are never overwritten; subsequent manifest and extraction must use the local snapshot.

cache-gitbook-llms / extract-markdown-drafts — deterministic GitBook Markdown

uv run loop-apidoc cache-gitbook-llms --url https://example.gitbook.io/docs \\
  --sources ./sources --coverage ./work/url_sources/coverage.json
uv run loop-apidoc manifest --sources ./sources --url https://example.gitbook.io/docs --output ./work/manifest.json
uv run loop-apidoc extract-markdown-drafts --sources ./sources --manifest ./work/manifest.json --output ./work/markdown-api-facts.json

The GitBook command fetches llms.txt once and caches every safe, same-origin, entry-prefix .md URL under the source directory, with URL/SHA-256/timestamp sidecars. Index/collision errors write no partial corpus; page failures stay visible in coverage. The facts JSON contains only line-cited, explicit endpoint headings, labelled tables and fenced examples. It is an aid, not extraction JSON or source authority; agents still read sources and run verify-extraction.

assess-sources — pre-extraction source quality gate

uv run loop-apidoc assess-sources --sources ./sources --manifest <manifest.json> \
  --observations <observations.json> --source-set <name> --output ./work/source-quality [--base-manifest <old manifest>]

Before extraction, assesses source quality against traceable observations, producing source-quality-report.{json,zh-TW.md} and a source diff source-diff.{json,md}, with a verdict of pass or reject. Pass the report directory to assemble --source-quality: reject aborts assembly before the run-dir is created; a passing report and source diff are retained in the run-dir for audit.

record-fingerprint / check-freshness / check-freshness-batch — scheduled source freshness gate

uv run loop-apidoc record-fingerprint --run-dir ./output/<run-id> --output ./work/source-fingerprint.json [--force]

uv run loop-apidoc check-freshness --fingerprint ./work/source-fingerprint.json \
  [--sources ./sources] [--json] [--report-dir ./work/freshness]

Once a run has completed or been approved as a Foundry asset, first use record-fingerprint to write a baseline source-fingerprint.json from that run directory (SHA-256 of local sources, plus one version-signal fetch per URL source); an existing file is not overwritten unless --force is given. Afterwards, running check-freshness on a schedule (e.g. cron) cheaply determines whether the sources are worth re-extracting, without re-parsing the whole document or calling any model: OpenAPI URL sources compare info.version (an identical version is treated as unchanged even if the bytes differ), HTML compares ETag/Last-Modified first then falls back to body SHA-256, and local files compare SHA-256. Exit codes: 0 = unchanged (re-parse can be skipped), 1 = changed (re-run extraction), 2 = inconclusive (a source could not be fetched or read). With --report-dir it additionally writes freshness-report.{json,md}; without it, no file is written.

To sweep many docsets at once, use check-freshness-batch instead:

uv run loop-apidoc check-freshness-batch --watchlist ./work/freshness-watchlist.json \
  [--json] [--report-dir ./work/freshness]

freshness-watchlist.json lists multiple watch items, each with a label (identifying name), fingerprint (the source-fingerprint.json path relative to the watchlist file's directory), an optional sources (local source root, likewise relative to the watchlist file), and an optional run_dir (display only). The command runs the same comparison as check-freshness for every item; a fetch or read failure on a single item only marks that item as error and records the reason, without aborting the whole batch; but a malformed watchlist file (invalid JSON or off-schema) fails outright. Aggregate exit codes: 0 = all unchanged, 1 = any changed, 2 = any inconclusive or errored. With --report-dir it additionally writes freshness-scan.{json,md}; without it, no file is written.

verify-extraction — check that the extraction JSON meets the contract

uv run loop-apidoc verify-extraction --sources ./sources --extraction ./work [--url <URL> ...]

Applies to the agent-produced extraction directory the same input gate as assemble (schema, source citations, cross-file invariants, source facts, deferral phrases), but writes nothing and creates no run directory, making it a fast way to validate the extraction JSON before assembly.

The source-fact gate mechanically scans the manifest's Markdown sources for endpoint declarations, parameter tables and fenced example blocks, then matches them to the extraction by (METHOD, path). When a matched source section documents fields or examples that the extraction dropped, the run fails closed — a silent omission is not the same as "the source does not say so." Naming the field in missing satisfies the gate, so it only ever forces a source-grounded gap, never an invention. Field names are resolved through schema_ref into inventory.schemas transitively, so factoring a shared request body out into a common type counts as deduplication, not an omission.

A companion check rejects placeholder answers that defer the work, and it does so in two layers to avoid false positives. Phrases that explicitly name the extraction itself ("further extraction", "not yet extracted", 「需進一步擷取」) count anywhere in a value, since a real API description never discusses its own extraction. Generic placeholders ("TBD", "to be determined", 「待補」) count only when they are the entire field, because "amount to be determined at capture" is legitimate API prose. ASCII phrases match on word boundaries — CJK has none, so those stay substring matches. Without this check a run could finish as passed with empty artifacts.

Know the scope limit. The scan only recognises well-structured Markdown: headings, GFM tables with a separator row, and fenced code blocks. A source flattened into long single lines — an HTML-to-text dump, for instance — yields zero facts, and the gate is a silent no-op on it. So a clean gate exit is not by itself evidence of a complete extraction; it only proves nothing contradicted the facts that could be mechanically read. On unstructured sources, keep relying on review.html and human review.

assemble — assemble + validate from the agent-produced extraction JSON

uv run loop-apidoc assemble \
  --sources ./sources \
  --extraction ./work \
  --output ./output \
  [--url <URL> ...] [--source-quality ./work/source-quality] [--url-coverage <coverage.json>] [--extractor-model <model name>] [--architecture-mode legacy|shadow] [--json] [--score]

Does not extract; it only assembles the extraction directory the agent has already produced (inventory.json + endpoints/*.json, optionally integration.json) into outputs: manifest → plan → generate → validate. If you pass the --source-quality directory produced by assess-sources, a reject verdict aborts before the run-dir is created, while a passing report is retained in the run-dir; --url-coverage checks fetch coverage when there are URL sources (warning only). --json prints run_id, run_dir, review_html, ok, status, report, and toolchain to stdout for the agent to parse and drive the correction loop. The run directory also gets a run.json descriptor recording the toolchain (cli_version, extraction_contract_version, skill_version, model) so a run can later be attributed to the versions that produced it; --extractor-model is the extraction model name, passed explicitly by the agent — omit it and the field stays null, because the CLI never guesses it. Exit codes: 0 = validation PASS, 1 = validation FAIL, 2 = extraction input file error. Adding --score also writes score/score.{json,md} after assembly (see below); the score does not affect the exit codes above.

--architecture-mode shadow opts into a non-authoritative Core observation after the legacy validation report is written. Success writes source-set, evidence, runtime result, claims, contract, decision, workflow, events, and comparison JSON under <run-dir>/core/; failure writes core/error.json and a warning. Neither result changes legacy validation, score, approval, Foundry, ok/status, or exit codes. The default is legacy and creates no core/.

Shadow evidence is claim-level: each material claim path is connected by explicit_support, derived_support, contradicts, or insufficient to an exact fragment and source artifact. Core uses deterministic value, table-cell, structured-path, enum, and source-fact verification; runtime confidence does not decide status. A filename-only or whole-document legacy citation becomes insufficient and remains unverified. Inspect relationships.json and projections/provenance.json for the claim → relationship → fragment → artifact trace.

validate — validate an existing run directory

uv run loop-apidoc validate --output ./output/<run-id>

Runs the four validation categories — structure / completeness / consistency / no-speculation — over the run directory's outputs, and writes the report into <run-dir>/validation/. Returns 0 on pass, 1 when there are ERROR-severity issues.

score — grade a completed run directory

uv run loop-apidoc score --output ./output/<run-id> [--profile ci|review] [--min-score N] [--json]

Performs a deterministic documentation-quality score on a completed run directory, computing 0–100 from five weighted categories (openapi_validity / completeness / consistency / source_grounding / reviewability), written to <run-id>/score/score.{json,md}. --profile sets the score threshold (ci = 85, review = 70), which --min-score can override. Exit codes: 0 = meets threshold, 1 = below, 2 = input file missing or malformed. assemble --score produces the same report as a side effect, but does not affect assemble's exit code (still decided by validation).

diff — compare the version difference between two runs

uv run loop-apidoc diff --base ./output/<old-run> --head ./output/<new-run>

Compares two completed run directories, classifying differences by downstream impact into breaking / additive / changed / source_only. By default it writes to <new-run>/diff/report.{json,md}; --output can target another directory. The comparison scope covers openapi.yaml, integration-contract.json, provenance.json, validation/report.json, and manifest.json; the first version does not compare the Markdown guide or generated examples. Exit codes: 0 = complete, 2 = input run-dir missing files or malformed.

foundry — project-local asset governance for an API project

uv run loop-apidoc foundry [init|import|approve|list|current] --help

Manages the project-local .foundry/api/ asset layer: import a completed run as a docset candidate (import), approve it into a versioned, self-contained official asset (approve, with the prior version auto-retired), and use a deterministic current pointer so downstream always uses the latest approved version (current / list).

review — local comparison, handoff, and human approval

uv run loop-apidoc review --project ./my-api --docset payment --run ./output/<run-id>

Starts a single-user workbench on 127.0.0.1; it automatically imports the completed run when the candidate does not yet exist. It compares with Foundry's current asset, or opens a baseline review for the first version. The page presents validation, version differences, and subjective conclusions, then saves them as candidate review/decision.json handoff. Only a human approval updates current. Failed validation, differences, or unresolved handoffs do not lock approval, but the resulting asset is marked needs_follow_up. The workbench never calls a model or replaces deterministic validation.

Output structure

Each run uses its own run directory:

output/
└── <run-id>/                       # run-id format: %Y%m%dT%H%M%S.%fZ (includes microseconds to avoid same-second collisions)
    ├── run.json                    # run descriptor (status + toolchain versions)
    ├── manifest.json               # source manifest
    ├── extraction/                 # extraction audit trail (not re-runnable raw input)
    │   ├── queries.jsonl           # per-round query log
    │   └── answers/                # per-query responses <query_id>.txt
    ├── plan/
    │   └── normalization-plan.json      # machine-readable normalization plan
    ├── openapi.yaml                # OpenAPI 3.1
    ├── api-guide.zh-TW.md          # Traditional-Chinese integration guide
    ├── review.html                 # manual review page for generated artifacts (offline HTML)
    ├── provenance.json             # source traceability for each output item
    ├── integration-contract.json   # signature/encryption integration contract (when the source provides it)
    ├── examples/                   # per-endpoint curl / TypeScript / Python request examples (when produced)
    ├── handoff/                    # developer handoff aids (derived artifacts, not the contract source)
    │   ├── integration-tasks.md    # implementation order / execution config / blocker checklist
    │   ├── postman_collection.json # Postman v2.1 request-shape collection (importable)
    │   └── sdk-hints.json          # concise SDK/client generation hints (does not duplicate the schema)
    ├── preparation-report.json     # pre-generation readiness assessment (produced by assemble)
    ├── preparation-report.md
    ├── validation/
    │   ├── report.json
    │   └── report.md
    ├── source-quality/             # source quality audit retained when --source-quality is passed
    │   ├── source-quality-report.json
    │   ├── source-quality-report.zh-TW.md
    │   ├── source-diff.json
    │   └── source-diff.md
    ├── score/                      # documentation quality score (assemble --score or loop-apidoc score)
    │   ├── score.json
    │   └── score.md
    ├── core/                       # optional observational Core artifacts (--architecture-mode shadow)
    │   ├── source-set.json
    │   ├── evidence.json
    │   ├── runtime-result.json
    │   ├── claims.json
    │   ├── relationships.json
    │   ├── contract.json
    │   ├── decision.json
    │   ├── workflow.json
    │   ├── events.json
    │   ├── comparison.json
    │   └── projections/
    │       ├── openapi.json
    │       ├── review-data.json
    │       └── provenance.json
    └── diff/                       # version difference when compared against another run (loop-apidoc diff)
        ├── report.json
        └── report.md

Note: the agent-produced extraction input (inventory.json + endpoints/*.json + optional integration.json) lives in the working directory passed to --extraction, not in the run-dir; the run-dir's extraction/ holds only the audit trail.

Only content that both exists in the plan and is source-grounded makes it into the OpenAPI and Markdown. OpenAPI fields that are required but missing from the source are filled with a minimal legal placeholder and marked with x-loop-status: missing-source plus a provenance gap record; if that gap affects integrability, completeness validation still fails. When the source provides an error-code lookup table, openapi.yaml preserves the source-grounded error-code→meaning mapping losslessly via the x-loop-error-code-map extension field (since 0.9.2).

Validation rules summary

Category Content
Structure OpenAPI 3.1 validity; every endpoint must have a method, a path, and at least one response
Completeness Sources marked unverified, missing required fields, and manifest coverage gaps (unreadable sources, etc.) fail validation
Consistency The endpoint set and security names must agree across OpenAPI, Markdown, and provenance
No speculation Every output item must map to a provenance source; content with no source support is a violation

Validation classifies issues: OPENAPI_INVALID / OUTPUT_MISMATCH → fixable by regeneration; REQUIRED_INFO_MISSING → the agent re-reads the relevant sources to fill the gap; SOURCE_UNVERIFIED / SOURCE_CONFLICT / UNSUPPORTED_ASSERTION → unfixable (fail-closed, reported as a gap/conflict). Correction is agent-driven based on the assemble --json report (re-read sources, overwrite the extraction JSON, then re-run) rather than a loop built into the CLI.

Package structure

Package Responsibility
loop_apidoc/manifest/ source scanning and manifest building
loop_apidoc/agentcli/ assemble.py (assemble the agent-written extraction JSON → plan→generate→validate), input_schema.py/gate.py/verify.py (extraction input contract checks, i.e. verify-extraction; gate.py also aggregates the source-fact and deferral violations from source_facts/), extraction.py (convert inventory.json into per-stage plan answers), preprocess.py (PDF→md preprocessing, pymupdf4llm)
loop_apidoc/extraction/ models and utilities shared by the agent extraction (models, stages, questions, store, jsonblock)
loop_apidoc/plan/ normalization plan construction and source-match classification
loop_apidoc/generate/ OpenAPI / Markdown / review.html / provenance / examples / handoff generation (one of the file-I/O exits)
loop_apidoc/validate/ structure / completeness / consistency / no-speculation validation and reporting
loop_apidoc/run/ run-id generation, result/status models, writing the plan into the run directory
loop_apidoc/diff/ run-to-run version diff: read the artifacts, classify differences (breaking / additive / changed / source_only), write diff/report.{json,md}
loop_apidoc/preparation/ pre-generation readiness assessment; inside assemble, grades manifest / inventory / endpoints / plan into preparation-report.{json,md} (one of the file-I/O exits)
loop_apidoc/score/ deterministic documentation-quality score for a completed run-dir, computing 0–100 from five weighted categories, output to score/score.{json,md} (one of the file-I/O exits)
loop_apidoc/source_quality/ pre-extraction source quality assessment and source version-diff report (assess-sources); a passing report can be retained with the run-dir for audit
loop_apidoc/freshness/ scheduled source freshness gate (record-fingerprint / check-freshness / check-freshness-batch): baseline fingerprint writing and comparison, exit codes 0/1/2 to decide whether re-extraction is needed, optional freshness-report.{json,md}; check-freshness-batch compares each watchlist item and aggregates into freshness-scan.{json,md} (one of the file-I/O exits)
loop_apidoc/url_catalog.py / url_corpus.py bounded URL navigation index, page caching, and related candidates (catalog-url / select-url / cache-url-pages / cache-url-entry / normalize-html-snapshot / related-url-pages), letting the agent read web docs from local evidence. Navigation parsing skips pure UI controls (anchors with no target, role="button", dropdown toggles) so a UI label never becomes a source page
loop_apidoc/url_adapters.py resolve_fetch_url (pure, no I/O): maps a documentation URL to the representation that actually carries its body before any request is made. A HackMD note URL resolves to its /download raw-Markdown endpoint, so fetching the JavaScript shell never silently loses the source; everything else stays an HTML fetch
loop_apidoc/source_facts/ deterministic source-fact inventory behind the extraction gate: markdown.py (pure scan of endpoint declarations, parameter tables and fenced example blocks), collect.py (reads the manifest-named Markdown sources — this package's only read exit), gate.py (compares facts against the extraction JSON, resolving schema_ref into inventory.schemas transitively), deferral.py (rejects placeholder answers; two layers, word-boundary matching for ASCII). Wired into agentcli/gate.py, so both verify-extraction and assemble fail closed on a silent omission
loop_apidoc/foundry/ project-local .foundry/api/ asset governance: docsets, candidate import and asset approval, maintaining the current pointer and review.state
loop_apidoc/review/ local single-user review workbench: auto-import a candidate, compare current/baseline, persist structured handoff, and let Foundry update current only after explicit human approval
