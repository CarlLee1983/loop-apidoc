MEET LOOP-APIDOC

Turn messy technical docs into a manual anyone can read and trust

Vendor integration docs come in a mess of formats, and they're often incomplete. loop-apidoc works like a first-stage refinery: it distills tangled material into a clean, source-traceable standard spec — and never makes anything up.

First, one quick definition: what is an API?
Put simply, an API is the "interface" through which two pieces of software talk to each other; and API documentation is the manual that teaches you how to "connect" to the other system. When you integrate with a payment, logistics, or SMS service, for example, the provider hands you exactly this kind of document.

WHY YOU NEED IT

How hard is life without it?

When a team has to integrate with many external services at once, just "understanding the docs" is a battle in itself — and it hides risks you can't see.

Ten vendors, ten formats

One hands you a PDF, another drops a web page on you, another a pile of code files. Just "reading" them eats up huge amounts of time and patience.

The docs are often missing pieces

A key field is left out, or what happens on failure is never spelled out. The most dangerous part: to hit a deadline, someone just "fills in the blanks by guessing."

A wrong guess means an incident

One small wrong guess can turn into a real production incident after launch — failed payments, lost orders — and only then do you find out "the docs never actually said that."

THE AI-COLLABORATION ERA

Why does vibe coding need it even more?

Once you hand the implementation to a coding agent, it doesn't matter how fast the code is written — feed it the wrong spec and everything is wrong. The quality of the agent's output depends on the quality of the spec you give it.

Raw docs are a breeding ground for hallucination

Hand a PDF straight to an agent and, when something's missing, it fills the gap with "common conventions" — silently assuming OAuth or a standard error format. When you're integrating payments, that plausible-looking guess is the most expensive kind of bug.

Agents need machine-readable facts

The refined OpenAPI and integration contract are specs an agent can consume directly — cheaper and steadier than re-reading dozens of PDF pages every session, and every agent and every project reads the same single source of truth.

People sign off — they don't walk away

Vibe coding shifts the human role from "writing code line by line" to "signing off on specs and outputs." Only when every statement is traceable to its source and an offline review page ships alongside it can you dare to put your name on the agent's output.

HOW IT WORKS

How does it pull that off?

The whole process is like a diligent organizer, turning messy material into a clean result step by step.

1

Collect the sources

Feed it documents in any format — PDFs, web pages, ready-made spec files, even a single public URL will do.

2

Read it closely, section by section

It extracts the content one section at a time, like a diligent reader taking notes — but it only copies what the docs really say, never filling in from imagination.

3

Assemble it into a standard format

It reassembles the scattered information into a single, unified standard spec that both machines and people can read.

4

Check its own work

It automatically validates the result: if something's missing, it goes back and re-reads to fill it in; if it truly can't find it, it honestly marks it "missing / conflict" rather than forcing a fill.

5

Deliver the results

It produces a standard spec, a readable Traditional-Chinese manual, source traceability, and a quality-check report.

WHAT YOU GET

In the end, here's what you get

From a standard spec, a Traditional-Chinese manual, and source traceability to a quality report — plus a developer handoff pack you can start building from right away. Each artifact has a clear purpose, so engineers and managers alike get what they need.

Standard spec blueprint

An industry-standard format that any tool or engineer can pick up and use directly, with no need to reinterpret it each time.

Integration contract

More than a list of individual endpoints, it strings them into an executable transaction flow: call this first, get that back, then carry it into the next step — so engineers understand how the whole thing connects.

Source traceability

Quality-check report

WHY YOU CAN TRUST IT

Every statement is traceable to its source

Just as a paper cites its references, every field and every rule loop-apidoc produces can be traced back to which source document and which passage stated it.

When someone asks "where did this setting come from?", you can pull up the matching source passage right away.

VERSION DIFF · NEW FEATURE

New version? See at a glance what changed

The vendor updated the docs — but what actually changed in the new version? loop-apidoc can compare the old and new versions and tag every change with one of four "signal lights," so you can gauge the risk at a glance.

Breaking change breaking

Could "break" your existing integration — handle with special care and prioritize it.

Additive additive

A new feature or field was added, usually safe and not affecting existing integrations.

Content adjustment changed

Existing content was modified — not necessarily dangerous, but worth a minute to look at.

Source-only change source_only

Only the source document itself moved; the final spec didn't actually change, so you can relax.

QUALITY SCORE · NEW FEATURE

Not just "pass or fail" — it gives you a score

The check report tells you "what's missing and whether it passes"; the score goes a step further, converting document quality into a 0–100 number that's handy for tracking over time and comparing across versions. The score is only a dashboard — it never changes whether a document passes.

Scored across five dimensions

Spec validity, completeness, consistency, source grounding, and reviewability — five dimensions weighted together into one total score.

Two scoring perspectives

"Ship gate" is stricter and suited to automated pipelines; "manual review" is more lenient and suited to a self-check before submission. Same document, two viewpoints.

Visible progress

Every correction shows the score climbing, giving the team a clear target to work toward instead of vaguely saying "it's probably fine."

ASSET GOVERNANCE · NEW FEATURE

Don't let good docs scatter — file them in your own library

Once you've produced a result you're happy with, you can "shelve" it in the project's own document-asset library — approved and versioned — so everyone always uses the same "current version" and no one ever ends up with a stale old file.

1

Shelve it as a candidate

Import a finished result into the asset library, where it sits in the "candidate" area — not yet live.

2

Approve and version it

Once you've confirmed it's good, approve it, and the system copies it into a self-contained, version-numbered official asset — automatically retiring the previous version.

3

Always use the current version

Downstream just asks "which one should I use now?" and the system steadily points to the latest approved version, so everyone stays aligned on the same document.
