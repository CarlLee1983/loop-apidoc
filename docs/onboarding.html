loop-apidoc 是一條來源依據式的 API 文件 pipeline

它把各家廠商雜亂的串接文件(PDF / Markdown / HTML / OpenAPI JSON / 公開網址),轉成一組標準化、可追溯的產物。這份文件帶你看懂整條流程怎麼跑、以及為什麼這樣設計,讀完就能銜接 ARCHITECTURE.md 與 SKILL.md。

讀完要掌握

  • 來源文件是唯一真實來源,缺漏就標記缺漏。
  • agent 只負責擷取,CLI 後段負責可重現組裝。
  • 每次 run 都留下驗證、評分、差異與交接證據。
開始前,先對齊幾個會反覆出現的詞
OpenAPI
描述 REST API 的業界標準格式(本專案產出 OpenAPI 3.1 YAML),工具鏈與工程師都能直接吃。
agent
當前的 coding agent(Claude Code 或 Codex)本身;在本專案裡,它就是擷取引擎。
subagent
由主 agent 派出的唯讀小幫手,只負責「讀來源、回傳 JSON」,不寫檔。
run-dir
一次執行產出的輸出目錄,裝著該次跑出來的所有產物(OpenAPI、指南、provenance、報告)。
不可協商的核心不變式

來源文件是唯一的真實來源。

來源沒寫的東西,一律留 null 並記錄到 missing —— 絕不推論、絕不用 REST/OAuth 慣例補上。

驗證遇到缺漏會大聲失敗(fail loudly),而不是用猜測讓文件「看起來完整」。整個專案的每一個設計決策,都是為了守住這條線。

寧可標記「缺漏」,也不編造內容
Loop Engineering

四個閉環:來源、品質、版本、交接

loop-apidoc 的工程重點是讓來源、品質、版本與交接都能被機器重跑、被人核對、被 downstream 使用者接手,而不只是產一份文件。

Source-grounded loop

sources → extraction → provenance

所有輸出必須回到來源引用;來源沒寫的資訊留缺漏,不靠慣例補齊。

Quality loop

preparation · validate · score

preparation 在生成前標出風險,驗證阻擋錯誤,score 把完成品轉成可比較的品質訊號。

Version loop

run-dir → diff/report

兩次 run 能依 breaking/additive/changed/source_only 分類,讓版本更新有 downstream impact 判讀。

Handoff loop

integration-contract · examples · handoff/

OpenAPI 是契約,衍生的 Postman、SDK hints、任務清單與 examples 幫下一位工程師接手。

關鍵架構

兩段式:agent 擷取 ‖ 確定性 CLI 後段

這是你最需要先建立的心智模型。整條 pipeline 切成兩半,中間用兩個 JSON 檔當唯一交界,兩邊各自可獨立測試、獨立替換。

前段 · 不確定性

agent 擷取(Claude Code / Codex)

  1. 唯讀 subagent fan-out:主 agent 派出多個 subagent,每個只讀檔 / 搜尋 / 抓 URL,回傳結構化 JSON,從不寫檔。
  2. 主 agent 把回傳的 JSON 彙整、寫成交界檔。
  3. 依 SKILL.md 分段擷取(見下方十段),確保只記錄來源真正寫的內容。
唯一交界 (seam) inventory.json endpoints/*.json
後段 · 確定性純函式

assemble(deterministic CLI)

  1. 不擷取,只組裝 agent 寫出的 JSON。
  2. 依序跑 manifest → plan → preparation → generate → validate。
  3. 以 --json 回報 ok / run_dir / report,讓 agent 自行驅動修正。
為什麼這樣切? 把「需要語意理解、本質不確定」的擷取交給 agent,把「組裝、生成、驗證」這些必須可重現的步驟交給確定性程式碼。曾經有過用子行程 claude -p 擷取的 run-agent 模式與 NotebookLM 後端,皆已於 2026-06 退役,現在 agent-native 是唯一擷取路徑。
端到端資料流

一次完整執行,經過哪些階段

每一階段都有明確的輸入 / 輸出(產物)。灰色標記的是可選階段;preparation、score、diff 與 handoff 讓一次 run 從「生成」延伸成可審查、可比較、可交接的工程資產。

0preprocess 把 PDF 轉成高保真 markdown(pymupdf4llm),保留表格與巢狀結構。 可選 → sources_md/
1擷取 agent 唯讀 fan-out 讀來源、主 agent 寫出交界 JSON。 inventory.json · endpoints/*.json
2manifest 掃描本地來源與 URL,建立來源盤點。 manifest.json
3plan 把擷取結果規格化成計畫,並做來源比對分類。 plan/normalization-plan.json
4preparation 生成前評估 manifest、inventory、endpoint JSON 與 plan 的就緒度。 preparation-report.{json,md}
5generate 生成 OpenAPI、中文指南、人工核對頁、來源追溯與交接輔助產物。 openapi.yaml · api-guide.zh-TW.md · review.html · provenance.json · handoff/
6validate 結構 / 完整性 / 一致性 / 禁止推測四類檢查並出報告。 validation/report.{json,md}
7score 對完成的 run-dir 做品質評分,不改變 validate 的 pass/fail 語意。 可選 → score/score.{json,md}
8diff 比較兩個已完成 run-dir,依 downstream impact 分類差異。 可選 → diff/report.{json,md}
CLI 介面

六個指令,各司其職

CLI 進入點是 Typer(cli.py)。注意:擷取不在 CLI 裡 —— 那是 agent 的事;CLI 只負責後段與工具步驟。

preprocess

PDF → markdown 前處理(可選),供 agent 更易讀取。

manifest

掃描本地來源 + URL,建立 manifest.json。

assemble

組裝 agent 寫出的 JSON → plan → preparation → generate → validate,以 --json 回報。

退出碼:0 驗證 PASS · 1 驗證 FAIL · 2 擷取輸入檔錯誤(不留孤兒目錄)。

validate

對既有 run-dir 重跑驗證並出報告(不重新生成)。

score

讀取既有 run-dir 的 OpenAPI、provenance、manifest 與 validation report,產出品質評分。

退出碼:0 pass · 1 needs_attention / fail · 2 run-dir 輸入錯誤。

diff

比較兩個已完成 run-dir 的版本差異。

退出碼:0 完成 · 2 輸入 run-dir 缺檔或格式錯誤(DiffInputError)。

擷取分段

擷取拆成十段

為什麼分段?避免單一回答承載全部內容而失真。agent 依這十段逐步擷取,收斂成 inventory.json 與逐 endpoint 的 endpoints/*.json。

01來源盤點
02API 系統概覽與術語
03環境 / base URL / 版本
04驗證 / 授權 / 簽章
05Endpoint 清單
06逐 endpoint 細節(method / path / 參數 / req / resp / 範例)
07共用 schema / enum / 資料限制
08錯誤碼與失敗行為
09rate limit / timeout / retry / idempotency / webhook
10來源衝突、缺漏、無法確認事項
產物

一次 run 交付契約、證據與交接資產

核心產物提供 API 契約與來源依據;衍生產物提供品質判讀、版本比較與 downstream 工程交接。

標準規格

openapi.yaml

OpenAPI 3.1,工具鏈與工程師可直接取用,不必各自解讀來源。

繁中串接指南

api-guide.zh-TW.md

給人讀的 Traditional Chinese 指南,不必硬啃原文。

來源追溯

provenance.json

每項輸出都標明出自哪份來源、哪一段,可逐一交叉比對。

驗證報告

validation/report

列出哪裡完整、哪裡缺漏 / 衝突,品質有憑據可查。

品質評分

score/score.{json,md}

把 openapi_validity、completeness、consistency、source_grounding、reviewability 轉成可比較分數。

交接包

handoff/ · examples/

任務清單、Postman collection、SDK hints 與請求範例,協助串接工程師接手。

人工核對頁

review.html

離線可開的單檔 HTML,把範圍、來源、缺漏與 API 結構彙整成一頁,供人眼快速核對。

可追溯 ↔ 驗證對齊

禁止推測,靠 target 逐一對齊來把關

這是「絕不臆測」如何被機器強制執行的關鍵機制:provenance.json 的 target 字串,與 OpenAPI 位置一對一對齊。

no-speculation 檢查就在這些 target 上交叉比對:任何進入輸出的內容,都必須能追溯回一個具來源依據的計畫項目,否則視為違規。對齊的位置包含 paths.{path}.{method}、components.schemas.{name}、components.securitySchemes.{name}。

修正迴圈

驗證問題會被分類,而不是盲目重試

沒有 CLI 內建的修正迴圈。assemble 回報分類結果,修正由 agent 自行驅動:回頭重讀來源、覆寫對應 JSON、再跑一次,最多 3 輪。問題分兩類:

可修正(FIXABLE)

OPENAPI_INVALID · OUTPUT_MISMATCH

產物本身有誤 → agent 修正 JSON 後重新生成即可。

REQUIRED_INFO_MISSING

必要資訊缺漏 → agent 回頭重讀受影響來源,補齊那一塊。

不可修正(fail-closed)

SOURCE_UNVERIFIED · SOURCE_CONFLICT · UNSUPPORTED_ASSERTION

來源無法確認 / 彼此衝突 / 不被支援的斷言 → 不補寫,如實回報為缺漏或衝突。這正是核心不變式落地的地方。

版本差異

換版本時,依 downstream impact 分四類

diff 拿同一支 API 的新舊兩個已完成 run-dir 相比,看的是「這次改版會不會害到下游串接」,而不是「來源改了哪幾個字」。每一筆差異都被歸到下面四種 impact 之一,並依嚴重度由高到低排序,讓你一眼看出該不該擋、該通知誰。

破壞性 breaking

可能讓既有串接直接壞掉,風險最高。

例如:刪掉一個端點(operation removed)、既有欄位/參數改成必填(became required)、新增必填參數、拿掉某個 response 或 security scheme、schema 結構改變。

→ 上線前先擋,主動通知所有串接方調整。

新增 additive

純粹多了東西,通常向後相容。

例如:新增端點(operation added)、新增選填參數/欄位、新增 response 或 media type、新增 schema / security scheme。

→ 可安心採用,下游要不要用新功能自己決定。

內容調整 changed

既有內容改了,但不必然破壞相容性。

例如:參數/欄位的說明文字變更(description changed)、servers 位址更動、info 版本資訊更新、components 內未被引用的 schema 移除。

→ 讀一下改了什麼,確認對自己這端沒有隱性影響。

僅來源異動 source_only

來源或中繼資料動了,但最終 API 契約沒變。

例如:來源檔本身變更(manifest source changed)、provenance 追溯調整、validation / preparation 報告內容變動,但 OpenAPI 規格本身不受影響。

→ 可放心,規格未變;若在意可回頭看來源為何動。

比較範圍:OpenAPI 路徑·方法·參數·schema·security·webhook、integration-contract、provenance、validation 摘要與 manifest(第一版不比 Markdown guide 與 examples)。輸出寫到 diff/report.{json,md};exit code 0 完成、2 代表輸入 run-dir 缺檔或格式錯誤(DiffInputError)。

套件邊界

程式碼怎麼分包

原則:除了少數 I/O 出口,其餘模組皆為純函式,便於單元測試。修改時請守住這個邊界,讓每個 loop 都能獨立驗證。

套件職責
manifest/掃描本地來源 + 建立 manifest.json
agentcli/assemble.py(組裝 JSON → plan→preparation→generate→validate)、input_schema.py(交界處的 pydantic 驗證)、extraction.py(inventory → plan 答案)、preprocess.py(PDF→markdown)
extraction/共用 models + 工具(stages / questions / store / jsonblock)
plan/規格化計畫 + 來源比對分類
preparation/產生前就緒度評估 + preparation-report(report.py 寫檔)
generate/生成 OpenAPI / Markdown / review.html / provenance / examples / handoff(寫檔)
validate/結構 / 完整性 / 一致性 / 禁止推測檢查 + 報告
run/run-id 產生、result/status models、把計畫寫入 run-dir(寫檔)
score/讀取完成 run-dir,以五類權重產生 score/score.{json,md}(report.py 寫檔)
diff/loader.py / compare.py / models.py / report.py(report.py 寫檔)

標記列含檔案 I/O 出口:generate/、run/、preparation/report.py、score/report.py 與 diff/report.py 會寫檔,其餘維持純函式。

名詞速查

seam
前後兩段的唯一交界(inventory.json + endpoints/*.json),也是測試的切點。
manifest
來源盤點檔,記錄掃到哪些來源與 URL。
normalization-plan
把擷取結果規格化後的計畫,generate 與 validate 的共同輸入。
provenance
來源追溯;target 與 OpenAPI 位置逐一對齊。
preparation
生成前就緒度評估,在品質 loop 裡先標出阻塞與注意事項。
score
完成 run-dir 的品質分數,用於 CI 或 review profile 的後段判讀。
handoff
從 OpenAPI 與 integration-contract 衍生的工程交接包,不是新的契約來源。
fail-closed
遇到無法確認 / 衝突時,選擇「不輸出」而非「猜一個」。
run-dir
單次執行的輸出目錄,裝著該次全部產物。
