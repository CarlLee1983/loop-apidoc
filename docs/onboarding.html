loop-apidoc 是一條來源依據式的 API 文件 pipeline

它把各家廠商雜亂的串接文件(PDF / Markdown / HTML / OpenAPI JSON / 公開網址),轉成一組標準化、可追溯的產物。這份文件帶你看懂整條流程怎麼跑、以及為什麼這樣設計 —— 讀完你就能銜接 ARCHITECTURE.md 與 SKILL.md。

開始前,先對齊幾個會反覆出現的詞
OpenAPI
描述 REST API 的業界標準格式(本專案產出 OpenAPI 3.1 YAML),工具鏈與工程師都能直接吃。
agent
當前的 coding agent(Claude Code 或 Codex)本身 —— 在本專案裡,它就是擷取引擎。
subagent
由主 agent 派出的唯讀小幫手,只負責「讀來源、回傳 JSON」,不寫檔。
run-dir
一次執行產出的輸出目錄,裝著該次跑出來的所有產物(OpenAPI、指南、provenance、報告)。
不可協商的核心不變式

來源文件是唯一的真實來源。

來源沒寫的東西,一律留 null 並記錄到 missing —— 絕不推論、絕不用 REST/OAuth 慣例補上。

驗證遇到缺漏會大聲失敗(fail loudly),而不是用猜測讓文件「看起來完整」。整個專案的每一個設計決策,都是為了守住這條線。

寧可標記「缺漏」,也不編造內容
關鍵架構

兩段式:agent 擷取 ‖ 確定性 CLI 後段

這是你最需要先建立的心智模型。整條 pipeline 切成兩半,中間用兩個 JSON 檔當唯一交界 —— 兩邊各自可獨立測試、各自可獨立替換。

前段 · 不確定性

agent 擷取(Claude Code / Codex)

  1. 唯讀 subagent fan-out:主 agent 派出多個 subagent,每個只讀檔 / 搜尋 / 抓 URL,回傳結構化 JSON,從不寫檔。
  2. 主 agent 把回傳的 JSON 彙整、寫成交界檔。
  3. 依 SKILL.md 分段擷取(見下方十段),確保只記錄來源真正寫的內容。
唯一交界 (seam) inventory.json endpoints/*.json
後段 · 確定性純函式

assemble(deterministic CLI)

  1. 不擷取,只組裝 agent 寫出的 JSON。
  2. 依序跑 manifest → plan → generate → validate。
  3. 以 --json 回報 ok / run_dir / report,讓 agent 自行驅動修正。
為什麼這樣切? 把「需要語意理解、本質不確定」的擷取交給 agent,把「組裝、生成、驗證」這些必須可重現的步驟交給確定性程式碼。曾經有過用子行程 claude -p 擷取的 run-agent 模式與 NotebookLM 後端,皆已於 2026-06 退役 —— 現在 agent-native 是唯一擷取路徑。
端到端資料流

一次完整執行,經過哪些階段

每一階段都有明確的輸入 / 輸出(產物)。灰色標記的是可選階段。

0preprocess 把 PDF 轉成高保真 markdown(pymupdf4llm),保留表格與巢狀結構。 可選 → sources_md/
1擷取 agent 唯讀 fan-out 讀來源、主 agent 寫出交界 JSON。 inventory.json · endpoints/*.json
2manifest 掃描本地來源與 URL,建立來源盤點。 manifest.json
3plan 把擷取結果規格化成計畫,並做來源比對分類。 plan/normalization-plan.json
4generate 生成 OpenAPI、中文指南、人工核對頁與來源追溯。 openapi.yaml · api-guide.zh-TW.md · review.html · provenance.json
5validate 結構 / 完整性 / 一致性 / 禁止推測四類檢查並出報告。 validation/report.{json,md}
6diff 比較兩個已完成 run-dir,依 downstream impact 分類差異。 可選 → diff/report.{json,md}
CLI 介面

五個指令,各司其職

CLI 進入點是 Typer(cli.py)。注意:擷取不在 CLI 裡 —— 那是 agent 的事;CLI 只負責後段與工具步驟。

preprocess

PDF → markdown 前處理(可選),供 agent 更易讀取。

manifest

掃描本地來源 + URL,建立 manifest.json。

assemble

組裝 agent 寫出的 JSON → plan → generate → validate,以 --json 回報。

退出碼:0 驗證 PASS · 1 驗證 FAIL · 2 擷取輸入檔錯誤(不留孤兒目錄)。

validate

對既有 run-dir 重跑驗證並出報告(不重新生成)。

diff

比較兩個已完成 run-dir 的版本差異。

退出碼:0 完成 · 2 輸入 run-dir 缺檔或格式錯誤(DiffInputError)。

擷取分段

擷取拆成十段

為什麼分段?避免單一回答承載全部內容而失真。agent 依這十段逐步擷取,收斂成 inventory.json 與逐 endpoint 的 endpoints/*.json。

01來源盤點
02API 系統概覽與術語
03環境 / base URL / 版本
04驗證 / 授權 / 簽章
05Endpoint 清單
06逐 endpoint 細節(method / path / 參數 / req / resp / 範例)
07共用 schema / enum / 資料限制
08錯誤碼與失敗行為
09rate limit / timeout / retry / idempotency / webhook
10來源衝突、缺漏、無法確認事項
產物

一次執行交付四份產物

全部落在該次的 run-dir 裡,各自對應明確用途。

標準規格

openapi.yaml

OpenAPI 3.1,工具鏈與工程師可直接取用,不必各自解讀來源。

繁中串接指南

api-guide.zh-TW.md

給人讀的 Traditional Chinese 指南,不必硬啃原文。

來源追溯

provenance.json

每項輸出都標明出自哪份來源、哪一段,可逐一交叉比對。

驗證報告

validation/report

列出哪裡完整、哪裡缺漏 / 衝突,品質有憑據可查。

人工核對頁

review.html

離線可開的單檔 HTML,把範圍、來源、缺漏與 API 結構彙整成一頁,供人眼快速核對。

可追溯 ↔ 驗證對齊

禁止推測,靠 target 逐一對齊來把關

這是「絕不臆測」如何被機器強制執行的關鍵機制:provenance.json 的 target 字串,與 OpenAPI 位置一對一對齊。

no-speculation 檢查就在這些 target 上交叉比對:任何進入輸出的內容,都必須能追溯回一個具來源依據的計畫項目,否則視為違規。對齊的位置包含 paths.{path}.{method}、components.schemas.{name}、components.securitySchemes.{name}。

修正迴圈

驗證問題會被分類,而不是盲目重試

沒有 CLI 內建的修正迴圈。assemble 回報分類結果,修正由 agent 自行驅動:回頭重讀來源、覆寫對應 JSON、再跑一次,最多 3 輪。問題分兩類:

可修正(FIXABLE)

OPENAPI_INVALID · OUTPUT_MISMATCH

產物本身有誤 → agent 修正 JSON 後重新生成即可。

REQUIRED_INFO_MISSING

必要資訊缺漏 → agent 回頭重讀受影響來源,補齊那一塊。

不可修正(fail-closed)

SOURCE_UNVERIFIED · SOURCE_CONFLICT · UNSUPPORTED_ASSERTION

來源無法確認 / 彼此衝突 / 不被支援的斷言 → 不補寫,如實回報為缺漏或衝突。這正是核心不變式落地的地方。

版本差異

換版本時,依 downstream impact 分四類

diff 比較兩個已完成 run-dir,涵蓋 OpenAPI 路徑·方法·參數·schema·security·webhook、integration-contract、provenance、validation 摘要與 manifest(第一版不比 Markdown guide 與 examples)。

破壞性 breaking

可能讓既有串接壞掉,需優先處理。

新增 additive

新功能 / 新欄位,通常向後相容。

內容調整 changed

既有內容有改動,值得確認影響。

僅來源異動 source_only

來源動了但最終規格沒變。

套件邊界

程式碼怎麼分包

原則:除了少數 I/O 出口,其餘模組皆為純函式,便於單元測試。修改時請守住這個邊界。

套件職責
manifest/掃描本地來源 + 建立 manifest.json
agentcli/assemble.py(組裝 JSON → plan→generate→validate)、input_schema.py(交界處的 pydantic 驗證)、extraction.py(inventory → plan 答案)、preprocess.py(PDF→markdown)
extraction/共用 models + 工具(stages / questions / store / jsonblock)
plan/規格化計畫 + 來源比對分類
generate/生成 OpenAPI / Markdown / provenance(寫檔)
validate/結構 / 完整性 / 一致性 / 禁止推測檢查 + 報告
run/run-id 產生、result/status models、把計畫寫入 run-dir(寫檔)
diff/loader.py / compare.py / models.py / report.py(report.py 寫檔)

標記列含檔案 I/O 出口:只有 generate/、run/ 與 diff/report.py 會寫檔,其餘維持純函式。
