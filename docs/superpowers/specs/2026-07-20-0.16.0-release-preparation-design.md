# loop-apidoc 0.16.0 Release Preparation Design

**Date:** 2026-07-20  
**Status:** Approved for implementation  
**Scope:** Prepare and commit the `0.16.0` release candidate; do not push or tag

## 1. Goal

Prepare the unreleased claim-level semantic evidence work on `main` as
`loop-apidoc 0.16.0`, with synchronized version metadata, complete bilingual release
documentation, corrected design-history status, and reproducible validation evidence.

The preparation ends at a clean, committed release candidate. It does not create or push
the `v0.16.0` tag.

## 2. Release Classification

This is a minor release because it adds backward-compatible user-visible behavior:

- typed exact evidence fragments;
- deterministic claim-path support verification;
- relationship-aware reconciliation and projections;
- new observational shadow artifacts, including `core/relationships.json` and
  `core/projections/`.

The default architecture mode remains `legacy`. The opt-in shadow path remains
non-authoritative and cannot change legacy validation, score, run status, Foundry state,
approval, publication, or CLI exit behavior.

## 3. Preparation Approach

Use the existing release workflow as the only version-metadata writer:

```bash
npm run release:prepare -- --version 0.16.0 \
  --summary "Add deterministic claim-level semantic evidence verification and exact-fragment provenance while preserving legacy authority."
```

The generated release-note skeleton is then completed manually. The release script
continues to own the synchronized version updates in:

- `pyproject.toml`;
- `loop_apidoc/__init__.py`;
- `.claude-plugin/plugin.json`;
- `uv.lock`;
- `README.md`;
- `README.en.md`;
- `docs/introduction.html`;
- `tests/test_plugin_manifest.py`.

No alternate script or hand-edited version path is introduced.

## 4. Documentation Corrections

Release preparation must also update the human-facing and historical documentation that
the version script does not govern:

1. `docs/RELEASE_NOTES_0.16.0.md`
   - English-primary summary and changed-behavior sections;
   - a Traditional-Chinese supporting summary;
   - exact validation results;
   - explicit disclosure that operator-provided benchmark sources were unavailable and
     strict-local validation remains a release-operator check.
2. `docs/superpowers/specs/2026-07-20-claim-level-semantic-support-design.md`
   - change status from proposed to implemented.
3. `docs/superpowers/specs/2026-07-20-core-shadow-integration-design.md`
   - preserve the historical whole-document design;
   - mark its evidence-granularity section as superseded by the claim-level semantic
     evidence design instead of rewriting history.
4. `docs/RELEASE_CHECKLIST.md`
   - update the benchmark-discovery wording from 10 cases to the current 11 cases.

The already-synchronized teaching and promotion documents need no behavioral rewrite
unless release verification finds a stale command, artifact name, or authority statement.

## 5. Benchmark Evidence and Waiver

The repository intentionally excludes operator-provided benchmark source snapshots.
Current CI-safe benchmark tests therefore discover all 11 cases but skip source-dependent
assertions when `benchmarks/<case>/sources/` is absent.

Release preparation must:

- run the deterministic CI-safe suite and quality gate;
- confirm all 11 benchmark cases are still discovered;
- retain the focused exact-fragment, verification, reconciliation, shadow-isolation, and
  projection tests as targeted evidence for the changed behavior;
- record the missing-source limitation in the release notes;
- leave `quality_gate.py --strict-local` and the manual artifact review unchecked for the
  release operator who has the dated source snapshots.

The preparation must not fetch substitute documents, fabricate benchmark success, or
weaken skip detection.

## 6. Validation Flow

Run the release checks after all metadata and documentation changes:

```text
release:prepare
→ complete release notes and documentation corrections
→ uv sync --dev
→ npm run tag:check
→ Ruff
→ full pytest with coverage ≥ 95%
→ CI-safe quality gate
→ focused documentation/release tests
→ git diff/status review
→ release preparation commit
→ v0.16.0 tag dry-run from the clean committed worktree
```

The release wrapper requires a clean worktree, so the tag dry-run runs after the release
preparation commit. It may fetch tags but must not push `HEAD` or create a tag.

## 7. Failure Handling

- A version synchronization or lock failure blocks preparation.
- A lint, test, coverage, quality-gate, release-documentation, or tag-ordering failure
  blocks the release commit until its root cause is corrected.
- Missing operator benchmark sources do not become a false pass. They are recorded as an
  explicit release-operator follow-up under the repository's documented historical-source
  exception.
- No failed or partial release preparation is tagged or pushed.

## 8. Commit Boundary

The final release preparation commit contains:

- synchronized `0.16.0` version metadata;
- completed `0.16.0` release notes;
- the design-history status corrections;
- the corrected 11-case release checklist;
- any narrowly scoped documentation fix exposed by validation.

It contains no product behavior change, no benchmark-source substitution, no generated
run directory, and no tag.

## 9. Acceptance Criteria

Release preparation is complete when:

1. every governed version location reports `0.16.0`;
2. the release notes accurately describe claim-level semantic evidence and legacy
   authority;
3. the two architecture design documents describe their implemented/superseded status
   without contradicting current teaching documentation;
4. the release checklist names 11 benchmark cases;
5. Ruff, the full coverage suite at or above 95%, the CI-safe quality gate, focused release
   documentation tests, and tag-order validation pass;
6. missing strict-local source validation is visible in the release notes;
7. the release candidate is committed on `main`;
8. no tag is created and nothing is pushed.
