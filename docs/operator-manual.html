Loop Engineering 的來源依據式（source-grounded）API 文件 pipeline

English version: README.en.md

loop-apidoc 是一套可重複執行的 CLI，將格式與完整度不一的 API 串接文件，整理成一致、可追溯的標準化產物：

  • OpenAPI 3.1 YAML（openapi.yaml）
  • 繁體中文 Markdown 串接文件（api-guide.zh-TW.md）
  • 離線人工核對頁（review.html）
  • 來源追溯資料（provenance.json）
  • 驗證與缺漏報告（validation/report.{json,md}）

核心原則:以來源文件為唯一事實依據。來源未提供的資訊一律不推測;必要資訊缺漏時,驗證會失敗並明確列出缺項,而非以慣例補寫。

運作方式

擷取引擎是當前的 coding agent 自己。在 Claude Code plugin 或 OpenAI Codex CLI 的 session 裡,agent 依 skills/loop-apidoc/SKILL.md 讀來源、以唯讀 subagent fan-out 擷取(每個 subagent 只讀檔與搜尋、回傳 JSON,不寫檔),主 agent 把結果寫成 inventory.json + endpoints/*.json,再呼叫確定性 CLI assemble 跑後段 plan → generate → validate。

CLI 本身不負責建立 session、登入私有網站或驅動瀏覽器 —— 它只在 agent 寫好擷取 JSON 後做組裝與驗證。

早期曾有以子行程 claude -p 擷取的 run-agent CLI 模式,已於 2026-06 退役(連同 NotebookLM 擷取後端一併移除);現在唯一擷取路徑是 agent-native。

完整流程

preprocess（可選） → URL 導航/快取（可選：catalog-url → select-url → cache-url-pages） → assess-sources 來源品質閘（可選） → 擷取（agent 唯讀 subagent fan-out） → verify-extraction → manifest → 規格化計畫 → 生成（OpenAPI + Markdown） → 驗證 → 修正（agent 驅動）

驗證失敗時,assemble 以 --json 回報分類後的問題;修正由 agent 自行驅動:依報告回頭重讀相關來源、覆寫對應的擷取 JSON,再重跑 assemble,最多 3 輪。三輪後仍失敗或遇到無法修正的缺漏／衝突,則以缺漏／衝突回報而不補寫。

在 OpenAI Codex CLI 使用

同一份 skill 也能在 Codex 執行。Codex 不會設 ${CLAUDE_PLUGIN_ROOT},因此把 CLI 裝成全域指令,並把 skill 掛進 Codex 的 skills 目錄:

# 1. 把 CLI 裝成全域 loop-apidoc 指令
uv tool install --from /path/to/loop-apidoc loop-apidoc

# 2. 把 skill 掛進 Codex（symlink 即可，改檔自動同步）
ln -s /path/to/loop-apidoc/skills/loop-apidoc ~/.codex/skills/loop-apidoc

SKILL.md 以 <APIDOC> 佔位符自動辨識環境:有 $CLAUDE_PLUGIN_ROOT 走 plugin 內含 CLI,否則退到全域 loop-apidoc。其餘流程(擷取 → assemble → 驗證 → 修正)兩邊一致。

安裝

需求:Python >=3.11,並使用 uv 管理環境。

# 安裝相依套件
uv sync

# 確認 CLI 可執行
uv run loop-apidoc --help

作為 Claude Code plugin 安裝後即可在 Claude session 內使用,CLI 由 plugin 內含;Codex 端見上方「在 OpenAI Codex CLI 使用」。

提供來源

agent 擷取前,操作者只需把來源備妥:

  1. 把本機來源檔案放進一個來源目錄(記為 <SOURCES>)。
  2. 公開 URL 直接以 --url 傳入,由 agent 以文字抓取。
  3. 表格密集或大型 PDF,建議先用 preprocess 轉成高保真 markdown(見下),再把擷取指向轉出目錄。

不需建立任何外部 Notebook、不需瀏覽器登入 —— 來源讀取由當前 agent 在 session 內完成。

支援的來源格式

PDF、Markdown、Microsoft Word、OpenAPI JSON／YAML、公開 URL。

使用方式

CLI 提供來源處理、URL evidence、草稿擷取、品質與擷取、來源新鮮度、run 處理、Foundry 資產治理，以及 review 本機人工覆核。日常擷取流程仍由 skill 驅動；每個命令都可單獨使用。

preprocess — PDF 轉高保真 markdown(可選)

uv run loop-apidoc preprocess --sources ./sources --out ./work/sources_md

以 pymupdf4llm 把 --sources 目錄下的每個 PDF 轉成保留表格與標題結構的 markdown(非 PDF 文字來源原樣複製)。--sources 也可指定單一檔案，此時只會處理該檔案。表格密集或大型 PDF 在擷取前先轉換,可避免原始 PDF 讀取扭曲表格;之後把擷取 subagent 指向 --out 目錄。

manifest — 建立來源 manifest

uv run loop-apidoc manifest --sources ./sources-or-file [--url <URL> ...] [--output manifest.json]

掃描本機來源,記錄相對路徑、格式、大小、SHA-256、掃描時間、是否受支援、重複判定與處理狀態;公開 URL 另記錄擷取時間、HTTP 狀態與內容雜湊。--sources 可接受目錄或單一來源檔案；若提供檔案，會以其父目錄作為 sources_root，且 manifest 僅包含該檔案。省略 --output 時輸出至 stdout。

URL 導航與快取 — catalog-url / select-url / cache-url-pages / related-url-pages

# 只下載入口頁一次，建立側欄導航索引（不自動擷取子頁）
uv run loop-apidoc catalog-url --url <入口頁 URL> --output ./work/url_sources/catalog.json

# 依明確範圍選取 catalog 節點（不下載任何 URL）
uv run loop-apidoc select-url --catalog ./work/url_sources/catalog.json --branch <分支> --term <關鍵詞> --output ./work/url_sources/selection.json

# 快取 catalog 全部頁面，建立本機正文／連結／實體索引（不呼叫模型）
uv run loop-apidoc cache-url-pages --catalog ./work/url_sources/catalog.json --output ./work/url_corpus

# 依正文連結與共享實體輸出候選頁卡片，模型只在需要時讀取候選頁正文
uv run loop-apidoc related-url-pages --corpus ./work/url_corpus/corpus.json --url <頁面 URL> --output ./work/candidates.json

token 節約式的網頁文件讀取流程:catalog.json 是完整的導航 coverage universe;corpus.json 不嵌入正文,只指向本機 raw/ 與 body/ 檔。若快取 HTML 是尚未渲染的 SPA shell，僅在同一 origin 探測四個固定路徑：/swagger.json、/openapi.json、/v3/api-docs、/api-doc/v3/sections；只有根欄位含 openapi 或 swagger 的 JSON 才另存為獨立 corpus source。探測失敗、非規格回應與一般 JSON 一律不記錄，命令只在 stderr 報告偵測到的 shell 數量，不會自動 headless rendering。catalog 為空或單頁文件時,用 cache-url-entry --url ... --output ... 直接快取入口頁;已下載的 HTML 可用 normalize-html-snapshot --input page.html --url ... --output sources/page.md 轉為受支援 Markdown,並寫出帶原始 URL 與 SHA-256 的 .source.json provenance sidecar。

snapshot-openapi-url — 固定直連 OpenAPI 來源

uv run loop-apidoc snapshot-openapi-url --url <OpenAPI JSON/YAML URL> \\
  --sources ./sources --coverage ./work/url_sources/coverage.json [--confirmed-by-user]

適用於 URL 已經直接回傳 Swagger 2.0 或 OpenAPI 3.x JSON／YAML 的情況。命令驗證規格宣告後，把原始位元組固定為本機來源，輸出 SHA-256，並寫入 method: direct 的 coverage ledger。既有快照與 coverage 不會被覆寫；後續 manifest 與擷取必須以本機快照為準。

cache-gitbook-llms / extract-markdown-drafts / scaffold-extraction — GitBook Markdown 工程化

uv run loop-apidoc cache-gitbook-llms --url https://example.gitbook.io/docs \\
  --sources ./sources --coverage ./work/url_sources/coverage.json
uv run loop-apidoc manifest --sources ./sources --url https://example.gitbook.io/docs --output ./work/manifest.json
uv run loop-apidoc extract-markdown-drafts --sources ./sources --manifest ./work/manifest.json --output ./work/markdown-api-facts.json
uv run loop-apidoc scaffold-extraction --sources ./sources --manifest ./work/manifest.json --output ./work/scaffold

此 GitBook 命令只抓一次 llms.txt，快取所有安全、同網域、入口路徑下的 .md，並保存 URL/SHA-256/時間 sidecar。索引或碰撞錯誤不會留下半套 corpus；單頁失敗保留為 coverage 的 fetch_failed。facts JSON 僅列出明示端點標題、標記表格與 fenced examples 的行號事實；scaffold-extraction 會把可機械取得的部分投影成 extraction-shaped JSON。兩者皆只供覆核：先把 scaffold 的 inventory.json 與 endpoints/ 複製進 workdir，重讀 citations、補齊 security/integration/missing，再對 workdir 跑 verify-extraction；./work/scaffold 不能直接當成 --extraction。

assess-sources — 擷取前來源品質閘

uv run loop-apidoc assess-sources --sources ./sources --manifest <manifest.json> \
  --observations <observations.json> --source-set <名稱> --output ./work/source-quality [--base-manifest <舊 manifest>]

在擷取前依可追溯 observation 評估來源品質,輸出 source-quality-report.{json,zh-TW.md} 與來源差異 source-diff.{json,md},結論為 pass 或 reject。把報告目錄傳給 assemble --source-quality:reject 會在建立 run-dir 前中止組裝;pass 的報告與來源差異會保存進 run-dir 供稽核。

record-fingerprint / check-freshness / check-freshness-batch — 來源新鮮度排程閘

uv run loop-apidoc record-fingerprint --run-dir ./output/<run-id> --output ./work/source-fingerprint.json [--force]

uv run loop-apidoc check-freshness --fingerprint ./work/source-fingerprint.json \
  [--sources ./sources] [--json] [--report-dir ./work/freshness]

當一份 run 已完成或已核准為 Foundry 資產後,先用 record-fingerprint 從該 run 目錄寫出基準 source-fingerprint.json(本機來源的 SHA-256、每個 URL 來源各抓一次的版本訊號);既有檔案不會被覆寫,除非加上 --force。之後排程(例如 cron)定期執行 check-freshness 即可低成本判斷來源是否值得重新擷取,不需重新解析整份文件也不呼叫模型:OpenAPI URL 來源比較 info.version(版本相同即使位元組不同也視為未變)、HTML 先比對 ETag／Last-Modified 再退回內文 SHA-256、本機檔案比對 SHA-256。退出碼:0=未變(可跳過重新解析)、1=已變(需重跑擷取)、 2=無法判定(有來源抓取或讀取失敗)。加上 --report-dir 時另外寫出 freshness-report.{json,md};未帶則不寫檔。

要一次巡檢多份 docset,改用 check-freshness-batch:

uv run loop-apidoc check-freshness-batch --watchlist ./work/freshness-watchlist.json \
  [--json] [--report-dir ./work/freshness]

freshness-watchlist.json 列出多筆巡檢項目,每筆含 label(識別名稱)、 fingerprint(相對於 watchlist 檔案所在目錄的 source-fingerprint.json 路徑)、選填的 sources(本地來源根目錄,同樣相對於 watchlist 檔案)與選填的 run_dir(僅供顯示)。命令對每一筆逐項執行與 check-freshness相同的比對,單一項目抓取或讀取失敗只會把該項標記為 error 並記錄原因,不會中止整批巡檢;但 watchlist 檔案本身格式錯誤(非合法 JSON 或不符 schema)會直接失敗。彙總退出碼: 0=全部未變、1=有任一已變、2=有任一無法判定或發生錯誤。加上 --report-dir 時另外寫出 freshness-scan.{json,md};未帶則不寫檔。

verify-extraction — 檢查擷取 JSON 是否符合契約

uv run loop-apidoc verify-extraction --sources ./sources --extraction ./work [--url <URL> ...]

對 agent 產出的擷取目錄套用與 assemble 相同的輸入閘(schema、來源引用、跨檔不變式、來源事實、延後語句與選填 v1 evidence[])。若提供 exact reference，必須能從 manifest source 依 typed locator 重新 materialize，並吻合 stated normalized-fragment SHA-256；其 claim_path 也必須在 shared normalized-plan projection 中解析成功。命令不寫檔、不建立 run 目錄,適合在組裝前快速驗證擷取 JSON。

來源事實閘會機械掃描 manifest 中的 Markdown 來源,取出端點宣告、參數表與圍籬範例區塊,再以 (METHOD, path) 與擷取結果對照。一旦對上的來源小節寫了欄位或範例、擷取卻交回空的,就 fail closed——靜默遺漏不等於「來源沒寫」。要主張來源沒寫,在 missing 裡具名該欄位即可通過,所以這道閘只會逼出有據可查的缺口,不會逼出捏造。欄位名會沿 schema_ref 遞迴解析進 inventory.schemas,因此把共用 request body 抽成共用型別算去重複,不算遺漏。

另一道檢查攔下佔位式延後答案,並分兩層以避免誤判。明確指涉「擷取這件事」的說法(further extraction、not yet extracted、「需進一步擷取」)出現在值的任何位置都算,因為真實 API 描述不會提到自己的擷取流程;泛用佔位字(TBD、to be determined、「待補」)則只有在整個欄位就只有這句時才算,因為「amount to be determined at capture」是合法的 API 描述。英文片語以詞界比對,CJK 沒有詞界可言,維持子字串比對。少了這道檢查,run 會以 passed 收場而產物是空的。

請注意適用範圍。這道掃描只認得結構良好的 Markdown:標題、含分隔列的 GFM 表格、圍籬程式碼區塊。若來源被壓成一行行超長文字(例如 HTML 轉純文字的傾印檔),掃描結果為零筆事實,這道閘就靜默失效。因此閘門乾淨通過本身並不等於擷取完整,它只證明「機械讀得到的事實」沒有被違反。面對非結構化來源,仍要靠 review.html 與人工核對。

assemble — 從 agent 產出的擷取 JSON 組裝 + 驗證

uv run loop-apidoc assemble \
  --sources ./sources \
  --extraction ./work \
  --output ./output \
  [--url <URL> ...] [--source-quality ./work/source-quality] [--url-coverage <coverage.json>] [--extractor-model <模型名稱>] [--architecture-mode legacy|shadow] [--json] [--score]

不擷取,只把 agent 已產出的擷取目錄(inventory.json + endpoints/*.json,選用 integration.json)組裝成輸出:manifest → plan → generate → validate。若傳入 assess-sources 產出的 --source-quality 目錄,reject 結論會在建立 run-dir 前中止,pass 報告則保存進 run-dir;--url-coverage 在有 URL 來源時檢核撈取涵蓋率(僅 warning)。--json 會把 run_id、run_dir、review_html、ok、status、report、toolchain 印到 stdout,供 agent 解析並驅動修正迴圈。run 目錄另寫出 run.json,記錄 toolchain(cli_version、extraction_contract_version、skill_version、model)以便日後歸因;--extractor-model 由 agent 明確帶入擷取模型名稱,省略即 null(CLI 不推測)。退出碼:0=驗證 PASS、1=驗證 FAIL、2=擷取輸入檔錯誤。加上 --score 會在組裝後一併寫出 score/score.{json,md}(見下),評分不影響上述退出碼。

--architecture-mode shadow 會在 legacy validation report 寫出後啟用非權威的 Core 觀測旁路。成功時將 source-set、evidence、runtime result、claims、contract、decision、workflow、events 與 comparison JSON 寫入 <run-dir>/core/；失敗時寫出 core/error.json 並顯示 warning。兩者都不改變 legacy validation、score、approval、Foundry、ok/status 或退出碼。預設為 legacy，不建立 core/。

Shadow evidence 提升為 claim-level：每個 material claim path 透過 explicit_support、derived_support、contradicts 或 insufficient 連到 exact fragment 與 source artifact。Core 使用確定性的值、table cell、structured path、enum 與 source-fact verification；runtime confidence 不決定 status。只有檔名或整份文件的 legacy citation 會成為 insufficient 並維持 unverified。請從 relationships.json 與 projections/provenance.json 檢查 claim → relationship → fragment → artifact 追蹤鏈。

validate — 驗證既有 run 目錄

uv run loop-apidoc validate --output ./output/<run-id>

對 run 目錄輸出執行結構／完整性／一致性／禁止推測四類驗證,並將報告寫入 <run-dir>/validation/。通過回傳 0,有 ERROR 級問題回傳 1。

score — 為完成的 run 目錄評分

uv run loop-apidoc score --output ./output/<run-id> [--profile ci|review] [--min-score N] [--json]

對已完成 run directory 做確定性文件品質評分,五類加權(openapi_validity／completeness／consistency／source_grounding／reviewability)算出 0–100,寫入 <run-id>/score/score.{json,md}。--profile 決定分數門檻(ci=85、review=70),可用 --min-score 覆寫。退出碼:0=達門檻、1=未達、2=輸入缺檔或格式錯誤。assemble --score 會附帶產生同一份報告,但不影響 assemble 的退出碼(仍由 validation 決定)。

diff — 比較兩次 run 的版本差異

uv run loop-apidoc diff --base ./output/<old-run> --head ./output/<new-run>

比較兩個已完成 run directory,依 downstream impact 把差異分類為 breaking／additive／changed／source_only。預設寫入 <new-run>/diff/report.{json,md},可用 --output 指定其他目錄。比較範圍涵蓋 openapi.yaml、integration-contract.json、provenance.json、validation/report.json 與 manifest.json;第一版不比較 Markdown guide 或 generated examples。退出碼:0=完成、2=輸入 run-dir 缺檔或格式錯誤。

foundry — API 專案本地資產治理

uv run loop-apidoc foundry [init|import|approve|list|current] --help

管理專案本地 .foundry/api/ 資產層:把完成的 run 匯入為 docset candidate(import),核准為版本化、自足複本的正式 asset(approve,舊版自動退位),並以確定性的 current 指標讓下游永遠取用最新核准版(current／list)。

review — 本機比對、交棒與人工核准

uv run loop-apidoc review --project ./my-api --docset payment --run ./output/<run-id>

在 127.0.0.1 啟動單使用者工作台；若 candidate 尚未存在，會自動匯入完成的 run。它與 Foundry 的 current 比對，第一版則建立 baseline review；畫面顯示驗證結果、版本差異與可交棒的主觀結論，並把結論寫成 candidate 的 review/decision.json。只有人工點選核准才更新 current。驗證失敗、差異或未處理 handoff 不會鎖死核准，但正式資產會標記為 needs_follow_up。工作台不呼叫模型，也不取代確定性驗證。

輸出結構

每次執行使用獨立 run directory:

output/
└── <run-id>/                       # run-id 格式:%Y%m%dT%H%M%S.%fZ（含微秒，避免同秒衝突）
    ├── run.json                    # run 描述子（狀態 + toolchain 版本）
    ├── manifest.json               # 來源 manifest
    ├── extraction/                 # 擷取稽核軌跡（非可重跑的原始輸入）
    │   ├── queries.jsonl           # 每輪查詢紀錄
    │   └── answers/                # 各查詢回應 <query_id>.txt
    ├── plan/
    │   └── normalization-plan.json      # 機器可讀規格化計畫

驗證規則摘要

類別 內容
結構 OpenAPI 3.1 合法性;endpoint 必須有 method、path 與至少一個 response
完整性 標記 unverified 的來源、缺漏必要欄位、manifest 涵蓋缺口(不可讀來源等)會使驗證失敗
一致性 OpenAPI 與 Markdown／provenance 的 endpoint 集合與 security 名稱需一致
禁止推測 每個輸出項目須對應 provenance 來源;無來源支持的內容視為違規

驗證會將問題分類:OPENAPI_INVALID / OUTPUT_MISMATCH → 可由重新生成修正;REQUIRED_INFO_MISSING → agent 重讀相關來源補齊;SOURCE_UNVERIFIED / SOURCE_CONFLICT / UNSUPPORTED_ASSERTION → 無法修正(fail-closed,回報為缺漏／衝突)。修正由 agent 依 assemble --json 回報自行驅動(重讀來源、覆寫擷取 JSON 後重跑),而非由 CLI 內建迴圈。

Benchmark 品質閘：CI-safe 與 strict-local

Benchmark harness 有四層保證：已提交 fixture 清單、discovery 守門、來源支撐執行、strict-local 預檢。目前的 13 是唯一 case 目錄數，不是 pytest test item 數；同一 case 會供多個參數化測試使用。完整 contract 見 docs/BENCHMARK_VALIDATION_PLAN.md。

CI-safe 操作

uv run python scripts/quality_gate.py

CI-safe 模式不要求 operator 本機持有 gitignored benchmarks/<case>/sources/。它會執行 lint 與完整 pytest，因此會驗證 fixture discovery，以及 REQUIRED_BENCHMARK_CASES 和 committed fixture 的 exact parity。缺來源時，source-backed case assertions 會標成 SKIP；這只證明 case 已提交且可探索，不是 benchmark PASS。

Strict-local 操作

uv run python scripts/quality_gate.py --strict-local

只在同一台機器持有全部 13 份原始、具日期的來源快照時執行。預檢會先拒絕任何缺少或內容為空的 required sources/，再跑 source-backed benchmark module；只要 pytest 回報任一 skip，整個 quality gate 就失敗。通過表示 required/committed 完全一致、所有來源支撐斷言都實際執行，且零 skip。

若歷史來源無法合法或可重現地取得，請記錄缺少的快照，完成確定性 CI-safe 檢查，並對本次改動做合法的來源支撐 spot-check。不得用新版文件、合成 fixture 或 upstream error page 代替歷史證據，也不得宣稱 strict-local passed。

loop_apidoc/run/ run-id 產生、結果／狀態 models、將計畫寫入 run 目錄

loop_apidoc/preparation/ 產生前就緒度評估,於 assemble 內把 manifest／inventory／endpoints／plan 評成 preparation-report.{json,md}(檔案 I/O 出口之一)

loop_apidoc/foundry/ 專案本地 .foundry/api/ 資產治理:docset、candidate 匯入與 asset 核准,維護 current 指標與 review.state
loop_apidoc/review/ 本機單使用者 review 工作台：自動匯入 candidate、比對 current／baseline、保存結構化 handoff，並在人工明確核准後讓 Foundry 更新 current
