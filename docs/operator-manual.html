loop-apidoc

Loop Engineering 的來源依據式（source-grounded）API 文件 pipeline

English version: README.en.md

loop-apidoc 是一套可重複執行的 CLI，將格式與完整度不一的 API 串接文件，整理成一致、可追溯的標準化產物：

  • OpenAPI 3.1 YAML（openapi.yaml）
  • 繁體中文 Markdown 串接文件（api-guide.zh-TW.md）
  • 離線人工核對頁（review.html）
  • 來源追溯資料（provenance.json）
  • 驗證與缺漏報告（validation/report.{json,md}）

核心原則:以來源文件為唯一事實依據。來源未提供的資訊一律不推測;必要資訊缺漏時,驗證會失敗並明確列出缺項,而非以慣例補寫。

運作方式

擷取引擎是當前的 coding agent 自己。在 Claude Code plugin 或 OpenAI Codex CLI 的 session 裡,agent 依 skills/loop-apidoc/SKILL.md 讀來源、以唯讀 subagent fan-out 擷取(每個 subagent 只讀檔與搜尋、回傳 JSON,不寫檔),主 agent 把結果寫成 inventory.json + endpoints/*.json,再呼叫確定性 CLI assemble 跑後段 plan → generate → validate。

CLI 本身不負責建立 session、登入私有網站或驅動瀏覽器 —— 它只在 agent 寫好擷取 JSON 後做組裝與驗證。

早期曾有以子行程 claude -p 擷取的 run-agent CLI 模式,已於 2026-06 退役(連同 NotebookLM 擷取後端一併移除);現在唯一擷取路徑是 agent-native。

完整流程

preprocess（可選） → 擷取（agent 唯讀 subagent fan-out） → manifest → 規格化計畫 → 生成（OpenAPI + Markdown） → 驗證 → 修正（agent 驅動，最多 3 輪）

驗證失敗時,assemble 以 --json 回報分類後的問題;修正由 agent 自行驅動:依報告回頭重讀相關來源、覆寫對應的擷取 JSON,再重跑 assemble,最多 3 輪。三輪後仍失敗或遇到無法修正的缺漏／衝突,則以缺漏／衝突回報而不補寫。

在 OpenAI Codex CLI 使用

同一份 skill 也能在 Codex 執行。Codex 不會設 ${CLAUDE_PLUGIN_ROOT},因此把 CLI 裝成全域指令,並把 skill 掛進 Codex 的 skills 目錄:

# 1. 把 CLI 裝成全域 loop-apidoc 指令
uv tool install --from /path/to/loop-apidoc loop-apidoc

# 2. 把 skill 掛進 Codex（symlink 即可，改檔自動同步）
ln -s /path/to/loop-apidoc/skills/loop-apidoc ~/.codex/skills/loop-apidoc

SKILL.md 以 <APIDOC> 佔位符自動辨識環境:有 $CLAUDE_PLUGIN_ROOT 走 plugin 內含 CLI,否則退到全域 loop-apidoc。其餘流程(擷取 → assemble → 驗證 → 修正)兩邊一致。

安裝

需求:Python >=3.11,並使用 uv 管理環境。

# 安裝相依套件
uv sync

# 確認 CLI 可執行
uv run loop-apidoc --help

作為 Claude Code plugin 安裝後即可在 Claude session 內使用,CLI 由 plugin 內含;Codex 端見上方「在 OpenAI Codex CLI 使用」。

提供來源

agent 擷取前,操作者只需把來源備妥:

  1. 把本機來源檔案放進一個來源目錄(記為 <SOURCES>)。
  2. 公開 URL 直接以 --url 傳入,由 agent 以文字抓取。
  3. 表格密集或大型 PDF,建議先用 preprocess 轉成高保真 markdown(見下),再把擷取指向轉出目錄。

不需建立任何外部 Notebook、不需瀏覽器登入 —— 來源讀取由當前 agent 在 session 內完成。

支援的來源格式

PDF、Markdown、Microsoft Word、OpenAPI JSON／YAML、公開 URL。

使用方式

CLI 暴露六個指令:preprocess、manifest、assemble、validate、score、diff。日常擷取流程由 skill 驅動(主要呼叫 preprocess 與 assemble);manifest、validate、score 與 diff 可獨立使用。

preprocess — PDF 轉高保真 markdown(可選)

uv run loop-apidoc preprocess --sources ./sources --out ./work/sources_md

以 pymupdf4llm 把 --sources 下的每個 PDF 轉成保留表格與標題結構的 markdown(非 PDF 文字來源原樣複製)。表格密集或大型 PDF 在擷取前先轉換,可避免原始 PDF 讀取扭曲表格;之後把擷取 subagent 指向 --out 目錄。

manifest — 建立來源 manifest

uv run loop-apidoc manifest --sources ./sources [--url <URL> ...] [--output manifest.json]

掃描本機來源,記錄相對路徑、格式、大小、SHA-256、掃描時間、是否受支援、重複判定與處理狀態;公開 URL 另記錄擷取時間、HTTP 狀態與內容雜湊。省略 --output 時輸出至 stdout。

assemble — 從 agent 產出的擷取 JSON 組裝 + 驗證

uv run loop-apidoc assemble \
  --sources ./sources \
  --extraction ./work \
  --output ./output \
  [--url <URL> ...] [--json] [--score]

不擷取,只把 agent 已產出的擷取目錄(inventory.json + endpoints/*.json)組裝成輸出:manifest → plan → generate → validate。--json 會把 run_id、run_dir、ok、status、report 印到 stdout,供 agent 解析並驅動修正迴圈。退出碼:0=驗證 PASS、1=驗證 FAIL、2=擷取輸入檔錯誤。加上 --score 會在組裝後一併寫出 score/score.{json,md}(見下),評分不影響上述退出碼。

validate — 驗證既有 run 目錄

uv run loop-apidoc validate --output ./output/<run-id>

對 run 目錄輸出執行結構／完整性／一致性／禁止推測四類驗證,並將報告寫入 <run-dir>/validation/。通過回傳 0,有 ERROR 級問題回傳 1。

score — 為完成的 run 目錄評分

uv run loop-apidoc score --output ./output/<run-id> [--profile ci|review] [--min-score N] [--json]

對已完成 run directory 做確定性文件品質評分,五類加權(openapi_validity／completeness／consistency／source_grounding／reviewability)算出 0–100,寫入 <run-id>/score/score.{json,md}。--profile 決定分數門檻(ci=85、review=70),可用 --min-score 覆寫。退出碼:0=達門檻、1=未達、2=輸入缺檔或格式錯誤。assemble --score 會附帶產生同一份報告,但不影響 assemble 的退出碼(仍由 validation 決定)。

diff — 比較兩次 run 的版本差異

uv run loop-apidoc diff --base ./output/<old-run> --head ./output/<new-run>

比較兩個已完成 run directory,依 downstream impact 把差異分類為 breaking／additive／changed／source_only。預設寫入 <new-run>/diff/report.{json,md},可用 --output 指定其他目錄。比較範圍涵蓋 openapi.yaml、integration-contract.json、provenance.json、validation/report.json 與 manifest.json;第一版不比較 Markdown guide 或 generated examples。退出碼:0=完成、2=輸入 run-dir 缺檔或格式錯誤。

輸出結構

每次執行使用獨立 run directory:

output/
└── <run-id>/                    # run-id 格式:%Y%m%dT%H%M%SZ
    ├── manifest.json            # 來源 manifest
    ├── extraction/
    │   ├── inventory.json       # API 盤點（agent 擷取產出）
    │   └── endpoints/           # 逐 endpoint 擷取 JSON
    ├── plan/
    │   └── normalization-plan.json   # 機器可讀規格化計畫
    ├── openapi.yaml             # OpenAPI 3.1
    ├── api-guide.zh-TW.md       # 繁體中文串接文件
    ├── review.html              # 生成產物人工核對頁（離線 HTML）
    ├── provenance.json          # 每個輸出項目的來源追溯
    ├── preparation-report.json  # 產生前就緒度評估（assemble 產出）
    ├── preparation-report.md
    ├── validation/
    │   ├── report.json
    │   └── report.md
    ├── score/                   # 文件品質評分（assemble --score 或 loop-apidoc score）
    │   ├── score.json
    │   └── score.md
    └── diff/                    # 與另一個 run 比較版本差異時（loop-apidoc diff）
        ├── report.json
        └── report.md

只有同時存在於計畫、且具來源依據的內容,才會進入 OpenAPI 與 Markdown。OpenAPI 必填但來源缺失的欄位,會以最小合法占位填入,並標記 x-loop-status: missing-source 與 provenance 缺漏紀錄;若該缺漏影響可串接性,完整性驗證仍會失敗。

驗證規則摘要

類別 內容
結構 OpenAPI 3.1 合法性;endpoint 必須有 method、path 與至少一個 response
完整性 標記 unverified 的來源、缺漏必要欄位、manifest 涵蓋缺口(不可讀來源等)會使驗證失敗
一致性 OpenAPI 與 Markdown／provenance 的 endpoint 集合與 security 名稱需一致
禁止推測 每個輸出項目須對應 provenance 來源;無來源支持的內容視為違規

驗證會將問題分類:OPENAPI_INVALID / OUTPUT_MISMATCH → 可由重新生成修正;REQUIRED_INFO_MISSING → agent 重讀相關來源補齊;SOURCE_UNVERIFIED / SOURCE_CONFLICT / UNSUPPORTED_ASSERTION → 無法修正(fail-closed,回報為缺漏／衝突)。修正由 agent 依 assemble --json 回報自行驅動(重讀來源、覆寫擷取 JSON 後重跑),而非由 CLI 內建迴圈。

套件結構

套件 職責
loop_apidoc/manifest/ 來源掃描與 manifest 建立
loop_apidoc/agentcli/ assemble.py(組裝 agent 寫出的擷取 JSON → plan→generate→validate)、extraction.py(把 inventory.json 轉成 plan 各 stage 答案)、preprocess.py(PDF→md 前處理,pymupdf4llm)
loop_apidoc/extraction/ agent 擷取共用的 models 與工具(models、stages、questions、store、jsonblock)
loop_apidoc/plan/ 規格化計畫建構與來源比對分類
loop_apidoc/generate/ OpenAPI / Markdown / review.html / provenance 生成(檔案 I/O 出口之一)
loop_apidoc/validate/ 結構／完整性／一致性／禁止推測驗證與報告
loop_apidoc/run/ run-id 產生、結果／狀態 models、將計畫寫入 run 目錄
loop_apidoc/diff/ run 對 run 版本差異:讀取產物、分類差異(breaking／additive／changed／source_only)、輸出 diff/report.{json,md}
loop_apidoc/preparation/ 產生前就緒度評估,於 assemble 內把 manifest／inventory／endpoints／plan 評成 preparation-report.{json,md}(檔案 I/O 出口之一)
loop_apidoc/score/ 對完成 run-dir 做確定性文件品質評分,五類加權算出 0–100,輸出 score/score.{json,md}(檔案 I/O 出口之一)
